# 发票中枢（invoice_hub）产品诊断报告

> 日期：2026-07-23 ｜ 视角：产品管理（用户价值 / 闭环 / 指标 / 优先级）
> 方法：基于代码与文档的事实分析。标注【事实】=代码已确认；【推断】=基于代码特征的合理推测，需与用户确认；【建议】=产品优化方向。

---

## 0. 一句话结论

**工程完成度高、产品成熟度早期。** 发票"抓取→解析→入库→查看→导出→模板回填"的主链路已经跑通，且工程上做了大量容错（IMAP 连接池、风控识别、断链续拉、OFD/PDF 择优选、增量水位线、对账自愈）。但作为"产品"，它在五件事上是明显的缺口：**没有用户画像、没有指标、核心价值（解析质量）不可观测、闭环最后一公里有断点、合规有硬伤**。

换句话说：东西能跑，但「为谁跑、跑得好不好、下一步往哪跑」这三道产品题目还没解。

## 2. 核心判断：先回答"为谁做的"【推断·需确认】

从功能指纹看（**多邮箱聚合 + 模板金额匹配回填**），最像的使用场景是：
- **代账 / 商务外包 / 人力服务公司**（如 FESCO 类）的会计，一个人管 N 个客户邮箱，把发票号回填到报销/台账模板；
- 其次是企业财务自管、个人报销归档。

但**整个项目（README + 代码 + docs）里没有任何用户画像、使用场景、用户故事文档**。这直接造成一个后果：所有功能的优先级都缺少"用户价值"这个锚点——代码层做得越多，越容易陷入"工程师觉得爽"而非"用户真的痛"。

> 这是第一个要和你拍板的问题：**这个工具主要是给谁用的？** 下面所有优先级都依赖于这个答案。

## 3. 分维度诊断

### 3.1 用户与场景（定位）【推断】
- **问题**：无画像、无场景文档；Onboarding 门槛高——第一步就要"各邮箱开启 IMAP 并生成授权码"，对非技术用户（大量会计非程序员）是硬坎，README 是写给开发者看的。
- **影响**：无法判断哪些是"高频刚需"、哪些是"工程师自嗨"。

### 3.2 核心闭环（价值流）【事实 + 建议】
完整：抓取 → 解析 → 入库 → 查看筛选 → 导出打包 ✅
断点：
1. **重复报销 / 发票查重缺失**【事实·高影响】——`invoices` 只按 `(account_id, invoice_no)` 去重，**不同买方**用同一张发票号报销不会被发现。对财务这是红线级需求（重复报销=资金损失+合规风险）。
2. **抓取失败不可见、不可一键自愈**【事实·中影响】——引擎内部有完善的重试/容错，但用户侧只在 Web 滚动日志里看到；腾讯风控、链接抓不到、PDF 下载失败，最终只落到一个 `note` 字段，**没有"待处理/失败队列"和重试入口**。
3. **模板未匹配无出口**【事实·中影响】——`matching.build_result` 有 `unmatched`，但没有"导出未匹配清单去人工认领"的闭环，财务拿着 90% 匹配的结果，剩下 10% 卡在系统里。
4. **下游断链**【建议设为 Non-goal】——回填后无法直连报销系统/电子税务局/ERP，最后一公里靠"下载 zip 人工上传"。

### 3.3 解析质量（核心价值地基）【事实·高影响】
- 发票解析是产品的**核心价值**，但目前**质量不可观测、不可信**：
  - 没有"解析置信度"概念：每条发票记录不知道 `amount`/`invoice_no`/`buyer` 各自可靠度；
  - 没有"解析质量报告"：本次抓取 N 张，其中多少张金额缺失、多少张发票号缺失、需人工核对多少张——`note` 存了"未识别到发票号"但不聚合、不推送；
  - OCR 依赖本机装 Tesseract，没装则扫描件直接放弃，用户无感知；
  - 对财务场景，**一张发票金额错/号错是大事**，而当前没有"先把不可信的挑出来给人看"的机制。

### 3.4 指标体系（产品盲点）【事实·高影响】
- **完全无埋点、无北极星、无成功率/准确率/召回率/使用频次**。产品处于"瞎跑"状态：
  - 不知道抓取成功率多少、解析命中率多少、模板匹配命中率多少；
  - 不知道用户每天用几次、高频操作是什么；
  - 无法做数据驱动的迭代决策（这正是产品通的原则：没有度量就没有改进）。

### 3.5 合规与隐私（硬伤）【事实·高影响】
- **邮箱授权码明文存储**（`accounts.password` 直接入库，仅前端脱敏展示）；
- **Web 控制台无鉴权**（绑定 127.0.0.1，但本机任何人/任何网页能访问即可读全部发票与税号）；
- 发票含**买方税号、金额**等敏感个人信息，无加密、无访问控制、无审计日志；
- 符合《个人信息保护法》风险：敏感个人信息处理需告知同意、最小必要、加密存储。

### 3.6 差异化与竞争【建议】
- 无任何竞品分析。市面已有：邮箱自带发票管家、微信/支付宝发票管家、财税 SaaS（畅捷通/金蝶/报销吧等）。本项目的潜在差异化是"**多邮箱聚合 + 本地隐私 + 模板回填自动化**"，但未被明确提炼成卖点。

## 6. Non-goals 建议（防范围蔓延）

- 不做发票**真伪查验**（对接税务局查验平台）——投入大、依赖外部，可后续独立立项；
- 不做报销/财务**下游系统对接**——除非问题 3 确认要做；
- 不做多用户/团队**权限 SaaS**——保持本地单用户定位，除非代账场景强烈需要（#9）；
- 不重造邮箱客户端——只做"发票"这一件事。

## 7. 北极星指标 & 指标体系建议（待埋点）

**北极星**：`已成功解析并可信入库的发票张数 / 月`（反映"真正帮用户搞定的发票量"，而非抓取动作量）

**驱动指标**：
- 抓取成功率 = 成功拉取邮件 / 命中发票关键词邮件
- 解析字段完整率 = 含发票号+金额+买方的发票 / 全部识别为发票的 PDF
- 模板匹配命中率 = 已匹配行 / 模板总行
- 重复报销拦截数

**健康指标**：
- 抓取失败率（风控/链接失效/PDF 下载失败）
- OCR 兜底占比（扫描件比例，提示安装 Tesseract）
- 单账号水位线停滞时长（是否某邮箱长期没新数据）

## 9. 场景修订（用户已确认真实使用场景 · 生效版）

> 用户于 2026-07-23 澄清：**不是代账/外包会计，而是工作场景下的"发票归集枢纽"**。
> 原报告第 2/4/5/9/10 节基于"代账"假设的排期，以此节为准作修订。

### 9.1 真实用户画像【事实·用户自述】
- **角色**：每天应对**多个供应商**的发票，做"归集 → 整理 → 汇总 → 打包给财务做账"的中枢角色；不是做账人，是**把发票从供应商手里收拢、归类、交给财务的人**。
- **当前手工流程（≈1–2 小时/天，机械）**：
  1. 逐个点开发票链接；
  2. **分好所属公司**（这张归 A 公司、那张归 B 公司）；
  3. 用软件识别「金额 + 发票号」；
  4. 回填到一份清单；
  5. 打包归集给财务。
- **目标**：用软件快速吃掉第 1–5 步的机械动作；**最终"skill 化"，让 agent 自动归集整理**。
- **技术能力**：本人偏技术、要自动化 → 非技术 Onboarding 不是首要痛点。

### 9.2 代码事实复核：系统缺"公司"这个维度【事实】
- `accounts` 表 = **邮箱账号**（IMAP/API），不是公司；
- `invoices` 表只有 `buyer` 一个**自动解析出的购买方名称字符串**，没有独立的"公司实体"；
- 因此"分好所属公司"这一步**完全靠脆弱的自动识别 `buyer` 字段硬扛**：
  - `buyer` 解析失败 → 归入 `(未知)` → 无法正确归类打包；
  - `buyer` 是简写/别名 vs 清单里的规范名 → 模板匹配易失配（`matching._buyer_match` 靠双向包含兜底，不稳）；
  - **没有任何"公司清单管理 + 人工归属/纠正"的机制**，你无法建立一张规范的公司表。
- 这是之前按"代账"假设时没看透的**最核心缺口**：你手工第 2 步（分所属公司）在系统里是"隐式 + 不可控"的。

### 9.3 修订后的价值流与断点
完整：抓取 → 解析 → 入库 →（隐式）按 buyer 归类 → 查看/筛选 → 导出打包 ✅
断点（依此场景）：
1. **公司归属缺失**（替换原 #9 多客户隔离）—— 见 9.2，P0。
2. **解析质量不可信**（原 #1）—— 仍是 P0，见 §3.3。
3. **按公司打包给财务的闭环缺失** —— 现有"模板回填 xlsx + 按 id 导出 zip"，但缺"按公司聚合 → 生成每家公司清单+PDF包"的一键产出。
4. **抓取失败/未归属无队列** —— 漏抓或没分到公司的发票，交给财务的包就不完整/卡在系统里。
5. **下游（给财务）仍是人工最后一公里** —— 但本场景下游就是"你自己/财务收包"，不是对接报销系统，故仍建议 Non-goal（见 §6）。

## 11. 修订版北极星 / 指标体系

**北极星**：`每天为用户省下的发票归集手工时长`（目标：1–2h → ≈0），辅以 `交给财务的归集包完整率`（不漏、不错）。

**驱动指标**：
- 抓取成功率 = 成功拉取 / 命中发票关键词邮件
- 解析字段完整率 = 含 发票号+金额+买方 的发票 / 全部识别为发票的 PDF
- **公司归属准确率** = 自动归属正确 / 全部需归属发票（新增，本场景核心）
- 打包完整率 = 已进包发票 / 应进包发票
- 重复发票拦截数

**健康指标**：
- 抓取失败率、未归属率（新增）
- OCR 兜底占比（扫描件比例）
- 单账号水位线停滞时长
